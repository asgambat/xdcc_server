# Fixes Implementati - 20/05/2026

## Problemi Risolti

### 1. ✅ Ricerca Imprecisa
**Problema**: Cercando "margo soldi 6" veniva trovato anche "margo soldi 3"  
**Causa**: Il filtro dei risultati non verificava che tutti i termini della query fossero presenti nel filename  
**Soluzione**: Aggiunta funzione `filterByQuery()` che verifica che tutti i termini della query siano presenti nel nome del file

**File modificati**:
- `internal/searchagg/filters.go`: Aggiunta funzione di filtro per query

### 2. ✅ Logging Dettagliato Mancante
**Problema**: Mancavano log dettagliati per troubleshooting delle operazioni IRC  
**Soluzione**: Aggiunto logging esteso per tutte le operazioni critiche:
- Connessione/disconnessione server IRC
- Invio query WHOIS e risposta
- Join/leave dei canali
- Invio messaggi al bot
- Inizio/fine sessione download

**File modificati**:
- `internal/irc/client.go`: Log per sessione download
- `internal/irc/handlers.go`: Log per operazioni IRC (WHOIS, JOIN, messaggi bot)

**Log che ora vengono stampati**:
```
=== Starting XDCC download session ===
Server: irc.rizon.net:6667
Total packs to download: 1
--- Starting pack download: file.mkv (pack #123) from bot BotName ---
→ Sending WHOIS query for bot: BotName
WHOIS response: bot is in channels: #channel
Joining channel: #channel
✓ Joined channel: #channel
→ Sending XDCC request to bot BotName: xdcc send #123
=== Closing IRC connection ===
```

### 3. ✅ Ordinamento Risultati
**Risolto precedentemente**: I risultati sono ordinati alfabeticamente per nome file

## Problemi Non Risolti (Richiede Refactoring Maggiore)

### ❌ Connessioni IRC Non Riutilizzate
**Problema**: I download creano sempre nuove connessioni IRC invece di riusare quelle gestite da `ircmanager`  
**Causa**: Il sistema attuale usa due architetture parallele:
- `internal/ircmanager`: Gestisce connessioni persistenti per la UI (sezione Servers)
- `internal/irc/client.go`: Crea connessioni temporanee per i download

**Architettura Attuale**:
```
Download → Queue Manager → Worker → irc.NewClient() → Nuova connessione IRC
                                                       ↓
                                                       Download
                                                       ↓
                                                       Disconnetti
```

**Architettura Desiderata**:
```
Download → Queue Manager → IRC Manager (connessioni persistenti)
                                       ↓
                                       Riusa connessione esistente
                                       ↓
                                       Download
                                       ↓
                                       Mantieni connessione
```

**Impatto**:
- I server usati per i download non appaiono nella sezione "Servers" della UI
- Se sei connesso manualmente a un server e parti un download, viene creata una connessione separata
- I canali joinati durante i download non vengono salvati/mostrati
- Maggior carico sui server IRC (connessioni multiple)

**Soluzione Richiesta** (complessità: ALTA):
1. Modificare `queue/worker.go` per usare `ircmanager` invece di `irc.NewClient()`
2. Estendere `ircmanager` con metodi per gestire download:
   - `ConnectForDownload(serverAddress, port) (connID, error)`
   - `JoinChannelForBot(connID, botName) (channelName, error)` - con WHOIS
   - `SendXDCCRequest(connID, bot, packMessage) error`
   - `WaitForDCC(connID, timeout) (dccInfo, error)`
3. Modificare `ircmanager/manager.go` per salvare i canali joinati nel DB
4. Aggiungere logica per mantenere le connessioni vive dopo i download
5. Gestire il caso in cui una connessione manuale esiste già

**Files da modificare**:
- `internal/queue/worker.go` - Integrazione con ircmanager
- `internal/ircmanager/manager.go` - Metodi per download
- `internal/queue/manager.go` - Passare ircmanager al worker
- `internal/api/api.go` - Condividere ircmanager tra queue e API
