## 1. Concorrenza e Goroutine

### 1.1 🔴 [ALTA] `handleSetupBootstrap` — Scrittura diretta a Config senza lock

**File**: `internal/api/handlers_system.go:588-613`

```go
func (a *API) handleSetupBootstrap(...) {
    cfg := a.Config
    cfg.IRC.Nickname = body.Nickname          // ← NO LOCK
    cfg.IRC.DefaultServers = []config.ServerConfig{...}  // ← NO LOCK
    cfg.Download.DestDir = body.DownloadDir   // ← NO LOCK
    cfg.UI.SetupCompleted = true              // ← NO LOCK
    // ...
    cfg.Save(a.ConfigPath)
}
```

**Problema**: L'handler accede direttamente ai campi di `Config` senza acquisire `mu.Lock()`, mentre altre goroutine (IRC manager, queue manager, SSE handler) leggono gli stessi campi tramite i getter thread-safe (`GetNickname`, `GetDownloadConfig`, ecc.). Questo è una **data race** che può causare letture parziali o inconsistenti, specialmente su architetture non-x86.

**Fix consigliato**: Usare `SnapshotAndApply` come tutti gli altri handler:

```go
oldCfg := a.Config.SnapshotAndApply(func(snap *config.Config) {
    if body.Nickname != "" {
        snap.IRC.Nickname = body.Nickname
    }
    // ... resto delle assegnazioni su snap
})
a.Config.ApplyPartial(a.ConfigPath, oldCfg)
```

### 1.2 🔴 [ALTA] `managedConnection.autoJoinChs` — Accesso concorrente senza protezione

**File**: `internal/ircmanager/manager.go:790, 1575-1586, 1625-1632`

```go
type managedConnection struct {
    autoJoinChs []string  // ← nessun mutex dedicato
    // ...
}
```

`autoJoinChs` è letto nell'handler `CONNECTED` (goroutine del girc, linea ~1050) e scritto in `joinChannel`/`leaveChannel` (goroutine HTTP handler, linee 1575-1586, 1625-1632). Questi accessi avvengono da goroutine diverse senza sincronizzazione. La lettura nell'handler CONNECTED itera la slice, e una scrittura concorrente potrebbe causare un panic o comportamento indefinito.

**Fix consigliato**: Proteggere `autoJoinChs` sotto `mc.mu` oppure usare un mutex separato. L'handler CONNECTED già acquisisce `mc.mu` per altri campi, quindi estendere la sezione critica per includere la lettura di `autoJoinChs` è la soluzione più semplice:

```go
// Nell'handler CONNECTED:
mc.mu.RLock()
channels := make([]string, len(mc.autoJoinChs))
copy(channels, mc.autoJoinChs)
mc.mu.RUnlock()
for _, ch := range channels {
    // ...
}
```

### 1.3 🟡 [MEDIA] `SnapshotAndApply` — Potenziale stallo con `mu.RLock` + `Replace`

**File**: `internal/config/config.go:1089-1103`

```go
func (c *Config) SnapshotAndApply(fn func(snap *Config)) *Config {
    c.mu.RLock()
    old := c.cloneLocked()
    post := c.cloneLocked()
    c.mu.RUnlock()   // ← rilascia RLock

    if fn != nil {
        fn(post)
    }
    c.Replace(post)   // ← acquisisce mu.Lock()
    return old
}
```

**Problema**: Tra il rilascio di `RLock` e l'acquisizione di `Lock` in `Replace`, un'altra goroutine potrebbe chiamare `SnapshotAndApply` concorrentemente, installando una diversa versione della config. La seconda chiamata "vince" e la prima sovrascrive le modifiche della seconda. Non è un deadlock, ma è una **lost update** (persa aggiornamento) nella finestra TOCTOU.

**Fix consigliato**: Per operazioni che devono essere serializzate, usare `mu.Lock()` direttamente invece di `RLock` + `Unlock` + `Lock`. Il costo aggiuntivo è trascurabile rispetto alla correttezza:

```go
func (c *Config) SnapshotAndApply(fn func(snap *Config)) *Config {
    c.mu.Lock()
    old := c.cloneLocked()
    post := c.cloneLocked()
    fn(post)
    // Copia diretta campo per campo (equivale a Replace ma senza Unlock+Lock)
    c.IRC = post.IRC
    c.HTTP = post.HTTP
    // ... ecc
    c.mu.Unlock()
    return old
}
```

### 1.4 🟡 [MEDIA] `Manager.Start` — RLock rilasciato e riacquisito in loop

**File**: `internal/ircmanager/manager.go:182-194`

```go
m.mu.RLock()
for _, s := range servers {
    if s.AutoConnect && s.Status == "disconnected" {
        if _, exists := m.conns[s.ID]; !exists {
            m.mu.RUnlock()          // ← rilascia
            m.ConnectServerByID(s.ID)  // ← acquisisce Lock internamente
            m.mu.RLock()            // ← riacquisisce
        }
    }
}
m.mu.RUnlock()
```

**Problema**: Questo pattern di rilascio-riacquisizione in un loop è fragile. Tra l'`RUnlock` e il successivo `RLock`, la slice `servers` potrebbe essere stata modificata (un server rimosso), causando accesso a dati non più validi o comportamento inaspettato. Inoltre, se `ConnectServerByID` fallisce lentamente, il loop continua con dati potenzialmente stantii.

**Fix consigliato**: Estrarre la lista degli ID da connettere sotto `RLock`, poi connetterli fuori dal lock:

```go
m.mu.RLock()
var toConnect []int64
for _, s := range servers {
    if s.AutoConnect && s.Status == "disconnected" {
        if _, exists := m.conns[s.ID]; !exists {
            toConnect = append(toConnect, s.ID)
        }
    }
}
m.mu.RUnlock()

for _, id := range toConnect {
    if err := m.ConnectServerByID(id); err != nil {
        m.logger.Warnf("connecting to server %d failed: %v", id, err)
    }
}
```

## 2. Goroutine Leak

### 2.1 🔴 [ALTA] `notifier.Manager.dispatch` — Goroutine fire-and-forget senza WaitGroup

**File**: `internal/notifier/notifier.go:781-791`

```go
func (m *Manager) dispatch(evt queue.Event) {
    for _, n := range m.notifiers {
        n := n
        go func() {       // ← goroutine lanciata senza tracking
            if err := n.Notify(evt); err != nil {
                m.logger.Errorf(...)
            }
        }()
    }
}
```

**Problema**: Ogni chiamata a `dispatch` lancia N goroutine (una per notifier) senza alcuna sincronizzazione. Se un notifier è lento (es. email SMTP con timeout di 15s) e arrivano molti eventi, queste goroutine si accumulano senza limiti. Inoltre, `drainEvents` chiama `dispatch` che lancia goroutine dopo il timeout di 100ms — queste goroutine potrebbero ancora essere in volo quando `Run` ritorna.

**Fix consigliato**: Usare un `sync.WaitGroup` o un semaphore per limitare la concorrenza, e attendere il completamento delle goroutine in `drainEvents`:

```go
func (m *Manager) dispatch(evt queue.Event) {
    var wg sync.WaitGroup
    for _, n := range m.notifiers {
        n := n
        wg.Add(1)
        go func() {
            defer wg.Done()
            if err := n.Notify(evt); err != nil {
                m.logger.Errorf(...)
            }
        }()
    }
    wg.Wait()  // oppure usare un semaphore per limitare la concorrenza
}
```

In alternativa, per non bloccare il loop principale, usare un worker pool con buffer limitato.

### 2.2 🟡 [MEDIA] `sendChannelGreeting` — Goroutine non tracciata nel WaitGroup

**File**: `internal/ircmanager/manager.go:1390`

```go
func (mc *managedConnection) sendChannelGreeting(cl *girc.Client, channel string) {
    go func() {
        // ... delay 2-4s, poi invia greeting
    }()
}
```

**Problema**: Questa goroutine è "fire-and-forget". Sebbene sia cancellata via `mc.ctx.Done()`, non è tracciata nel `mc.wg` WaitGroup della connessione. Se `disconnect()` viene chiamato subito dopo `sendChannelGreeting`, la goroutine potrebbe tentare di inviare un messaggio tramite un client IRC già chiuso, causando un panic o un errore silenzioso.

Anche se `activeGreetingClient` verifica che il client sia ancora valido, c'è una piccola finestra tra il controllo e l'invio effettivo.

**Fix consigliato**: Aggiungere la goroutine al WaitGroup:

```go
mc.wg.Add(1)
go func() {
    defer mc.wg.Done()
    // ... resto della logica
}()
```

### 2.3 🟡 [MEDIA] `DCC transfer goroutines` — Non tracciate nel WaitGroup del Client

**File**: `internal/irc/dcc.go:152-154`

```go
go c.ackSender()
go c.progressPrinter()
go c.receiveData()
```

**Problema**: Queste tre goroutine sono lanciate senza tracciamento. `receiveData` chiama `finishSuccess` o `finishWithError` che chiudono `downloadDone`, che a loro volta sbloccano `ackSender` e `progressPrinter`. Tuttavia, se il contesto viene cancellato mentre la goroutine `ackSender` è bloccata su `ackQueue`, e nessuno chiude `downloadDone` (perché il packState è stato rimpiazzato da `resetForPack`), la goroutine rimane bloccata permanentemente.

**Fix consigliato**: Aggiungere un `sync.WaitGroup` a `packState` e usare `defer wg.Done()` nelle tre goroutine. In `resetForPack`, attendere il completamento prima di rimpiazzare `c.ps`.

## 3. Memory e Resource Leak

### 3.1 🟡 [MEDIA] `SSE Hub.Publish` — Accumulo di client lenti senza backpressure

**File**: `internal/sse/hub.go:150-194`

```go
func (h *Hub) Publish(eventType string, payload map[string]interface{}) {
    // ...
    var dead []chan Event
    for ch := range h.clients {
        select {
        case ch <- evt:
        default:
            dead = append(dead, ch)  // ← eviction immediata
        }
    }
    // ...
}
```

**Problema**: I client lenti vengono evictati immediatamente quando il loro buffer di 256 eventi è pieno. Sebbene questo prevenga il blocco del publisher, un client che è temporaneamente lento (es. 1 secondo di ritardo) viene disconnesso e deve riconnettersi, causando un "thundering herd" di riconnessioni. Per applicazioni real-time questo può degradare significativamente l'esperienza utente.

**Fix consigliato**: Considerare un buffer più grande (es. 1024) oppure una strategia di "grace period" che permette al client di recuperare prima dell'evizione.

---

### 3.2 🟡 [MEDIA] `channellog.Logger` — File handle non chiusi per canali non più monitorati

**File**: `internal/channellog/channellog.go`

**Problema**: I file di log per canale vengono aperti lazily in `getOrOpen` e rimangono aperti per tutta la vita del `Logger`. Se un canale viene rimosso dalla lista `channel_log` nella configurazione, il file rimane aperto e continua a essere scritto (perché la verifica avviene solo nel chiamante, non nel Logger). Non c'è un meccanismo per chiudere i file dei canali non più monitorati.

**Fix consigliato**: Aggiungere un metodo `CloseChannel(channel string)` e chiamarlo quando la configurazione cambia. In alternativa, implementare un meccanismo di LRU eviction per i file meno recentemente usati.

### 3.3 🟢 [BASSA] `RateLimiter` — Entries non evictati per IP che non fanno più richieste

**File**: `internal/api/ratelimit.go:46-52`

La `evictStaleLocked` viene eseguita solo quando `now.Sub(rl.lastSweep) >= rl.window`. Se il window è 60s e il traffico è costante, la pulizia avviene ogni 60s. Ma se il traffico è burst-only (es. un picco di 1000 IP in 10 secondi, poi silenzio), quelle 1000 entries rimangono in memoria fino al prossimo burst.

**Fix consigliato**: Questo è un problema minore data la dimensione tipica delle entries. Per ambienti ad alto traffico, considerare una `sync.Map` o una cleanup più frequente.

### 4.2 🟡 [MEDIA] `disconnectServer` — Shutdown in due fasi con timeout complesso

**File**: `internal/ircmanager/manager.go:399-430`

```go
select {
case <-done:
    gracefulShutdown = true
case <-time.After(5 * time.Second):
    // ...
}
if !gracefulShutdown {
    select {
    case <-done:
    case <-time.After(10 * time.Second):
        // ...
    }
}
```

**Problema**: La logica di shutdown in due fasi (5s + 10s = 15s) è complessa e difficile da testare. Inoltre, la goroutine `go func() { conn.wg.Wait(); close(done) }()` rimane in volo anche dopo che il chiamante ritorna, potenzialmente chiudendo `done` su una goroutine già completata.

**Fix consigliato**: Usare `context.WithTimeout` e un singolo canale di completamento:

```go
shutdownCtx, cancel := context.WithTimeout(context.Background(), 15*time.Second)
defer cancel()
select {
case <-shutdownCtx.Done():
    return fmt.Errorf("server %d shutdown timeout", serverID)
case <-done:
    return nil
}
```

### 4.4 🟢 [BASSA] `entities.XDCCPack.MarshalJSON` — Alias pattern fragile

**File**: `internal/entities/pack.go:189-206`

```go
type Alias XDCCPack
return json.Marshal(&struct {
    // ...
    *Alias
}{Alias: (*Alias)(p)})
```

**Problema**: Il pattern `type Alias XDCCPack` con `*Alias` embedding funziona perché `sync.RWMutex` ha un metodo `MarshalJSON` zero-value che non produce output. Tuttavia, se `sync.RWMutex` cambiasse implementazione interna, questo pattern potrebbe rompersi silenziosamente. Inoltre, l'`Alias` copia il valore del mutex (anche se zero), che `go vet` potrebbe segnalare.

**Fix consigliato**: Creare un struct di output separato senza mutex, invece di usare l'Alias pattern.

## 5. Best Practices Go

### 5.1 🟡 [MEDIA] `context.Background()` usato per operazioni DB durante shutdown

**File**: `internal/ircmanager/manager.go:931,942`, `internal/queue/manager.go:971-974`

```go
_ = mc.manager.store.SetServerStatus(context.Background(), mc.id, "disconnected")
```

**Problema**: L'uso di `context.Background()` per operazioni DB significa che se SQLite è lento o bloccato, l'operazione può durare indefinitamente senza timeout. Durante lo shutdown, questo può impedire al processo di terminare.

**Fix consigliato**: Usare `context.WithTimeout(context.Background(), 5*time.Second)` per tutte le operazioni DB durante lo shutdown, come già fatto in `queue.Manager.Stop()`:

```go
shutdownCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
_ = mc.manager.store.SetServerStatus(shutdownCtx, mc.id, "disconnected")
```

### 5.2 🟡 [MEDIA] Errori ignorati sistematicamente con `_`

**File**: Multipli file, esempi:

- `internal/ircmanager/manager.go:1075`: `_ = mc.manager.store.SetChannelJoined(...)`
- `internal/queue/manager.go:388`: `_ = qm.store.RequeueDownload(...)`
- `internal/api/handlers_system.go:93`: `_, _ = w.Write([]byte(replaced))`

**Problema**: Sebbene per operazioni di "best effort" (update DB, write HTTP) sia accettabile ignorare l'errore, la pratica sistematica di `_` rende difficile il debugging quando qualcosa va storto. Almeno un log di warning dovrebbe essere prodotto.

**Fix consigliato**: Per le operazioni DB, loggare l'errore invece di ignorarlo silenziosamente:

```go
if err := mc.manager.store.SetChannelJoined(mc.ctx, channels.ID, true); err != nil {
    mc.manager.logger.Warnf("failed to mark channel %s as joined: %v", ch, err)
}
```

### 5.4 🟢 [BASSA] `pubsub.Hub` non ha meccanismo di eviction per subscriber lenti

**File**: `internal/pubsub/pubsub.go:86-92`


## 6. Performance

### 6.1 🟡 [MEDIA] `Config.GetDownloadConfig` — Copia dell'intera struct ad ogni accesso

**File**: `internal/config/config.go:956-960`

```go
func (c *Config) GetDownloadConfig() DownloadConfig {
    c.mu.RLock()
    defer c.mu.RUnlock()
    return c.Download  // ← copia per valore
}
```

**Problema**: `DownloadConfig` è una struct relativamente grande (contiene stringhe, int, ecc.). Ogni chiamata a `GetDownloadConfig` (che avviene frequentemente in `tryDispatch` e `startDownload`) esegue una copia completa della struct. Sebbene Go sia efficiente nelle copie di struct, per campi stringa questo comporta allocazione + copia.

**Fix consigliato**: Per i campi scalari usati più frequentemente (es. `MaxParallelTotal`, `ChannelJoinDelay`), aggiungere getter dedicati che restituiscono solo il valore richiesto sotto `RLock`, evitando la copia dell'intera struct.

---

### 6.2 🟢 [BASSA] `sse.Hub.Publish` — Lock globale per broadcast

**File**: `internal/sse/hub.go:150-194`

`Publish` acquisisce `h.mu` (write lock) per l'intera durata del broadcast a tutti i client. Se ci sono molti client (es. 100+), il tempo di hold del lock aumenta, bloccando `Subscribe` e `Unsubscribe`.

**Fix consigliato**: Per il carico tipico di questo progetto (pochi client SSE), questo è accettabile. Per scalabilità futura, considerare un design basato su canali dispatch (un canale per subscriber con goroutine dedicata).

---

### 6.3 🟢 [BASSA] `searchLive` — Semaforo manuale invece di `errgroup`

**File**: `internal/searchagg/aggregator.go:460-521`

L'implementazione usa un semaforo manuale (`sem := make(chan struct{}, 3)`) con `WaitGroup` per la concorrenza. Questo è funzionalmente corretto ma potrebbe beneficiare di `golang.org/x/sync/errgroup` per una gestione più idiomatica degli errori e del contesto.

## 7. Suggerimenti Architetturali

### 7.1 🟡 Interfaccia `IRCManagerInterface` — Accoppiamento con `entities.XDCCPack`

L'interfaccia `IRCManagerInterface` accetta `*entities.XDCCPack` come parametro, creando un accoppiamento stretto tra il queue manager e l'entità del dominio. Per migliorare la testabilità e la separazione delle responsabilità, considerare l'uso di un DTO (Data Transfer Object) o di un'interfaccia più generica.

### 7.2 🟡 Lifecycle Management — Pattern inconsistente tra componenti

I tre componenti principali (`ircmanager.Manager`, `queue.Manager`, `searchagg.Aggregator`) hanno pattern di lifecycle diversi:
- `ircmanager`: `Start()` + `Stop()` con `context.WithCancel`
- `queue.Manager`: `Start()` + `Stop()` con `context.WithCancel` + `done` channel + `closing` atomic
- `searchagg.Aggregator`: `Start(ctx)` + `Stop()` con canali separati per cleanup e stats

Considerare l'adozione di un'interfaccia `Lifecycle` comune:

```go
type Lifecycle interface {
    Start(ctx context.Context) error
    Stop()
    Done() <-chan struct{}
}
```

### 7.3 🟢 Config Hot-Reload — Nessun meccanismo per notificare i componenti dei cambiamenti

Quando la configurazione viene aggiornata via API (es. `handleUpdateConfig`), solo il rate limiter viene sincronizzato esplicitamente (`syncMsgRateLimiter`). Altri componenti (es. `ircmanager` che usa `GetNickname()`, `queue.Manager` che usa `GetDownloadConfig()`) leggono la config su ogni operazione, il che funziona grazie ai getter thread-safe. Tuttavia, cambiamenti come l'aggiornamento di `channel_blacklist` o `channel_log` richiederebbero una notifica ai componenti per prendere effetto immediato (es. chiudere i file di log dei canali rimossi).
