# Go Code Review — xdcc-go
**Data:** 24 maggio 2026  
**Scope:** Backend Go (`internal/` + `cmd/`)  
**Focus:** Concorrenza, goroutine leak, race condition, memory/resource leak, performance, idiomaticità Go.

---

## Executive Summary

Il codebase presenta un'architettura concorrente relativamente solida, con pattern maturi per la gestione di connessioni IRC persistenti, download paralleli e eventi SSE. Tuttavia, sono stati identificati **16 issue** di severità variabile che meritano attenzione, con particolare focus su:

- **2 issue ALTA severità**: potenziali goroutine leak e race condition in `ircmanager/manager.go`
- **5 issue MEDIA severità**: problemi di concorrenza, resource leak e code smell
- **9 issue BASSA severità**: miglioramenti idiomatici, performance e best practices

## 1. Concorrenza e Goroutine — Issue ALTA

### 1.1 [ALTA] Goroutine Leak in `ircmanager.Manager.run()` — `downloadWg` usato in modo errato

**File:** `internal/queue/manager.go` (riga ~83)  
**Problema:** `downloadWg sync.WaitGroup` viene dichiarato ma l'uso del WaitGroup è asimmetrico. Il `Add(1)` avviene in `startDownload`, ma il `Done()` è nel callback di completamento `completeFn` che è chiamato all'interno della goroutine worker. Se il worker non arriva mai a chiamare `completeFn` (es. panic, ctx cancellato prima del callback), il `downloadWg` rimane bloccato.

Inoltre, il metodo `Wait()` del downloadWg è usato in `Shutdown()` (riga ~320), ma se un download è in corso e il context viene cancellato, il worker potrebbe non chiamare `completeFn`, causando un deadlock infinito in `Shutdown()`.

**Fix consigliato:**
- Usare un `doneCh chan struct{}` per ogni download, oppure
- Aggiungere un `defer qm.downloadWg.Done()` immediatamente dopo l'`Add(1)`, e usare il callback `completeFn` solo per segnalare completamento logico, non per il conteggio del WaitGroup.

```go
// In startDownload:
qm.downloadWg.Add(1)
go func() {
    defer qm.downloadWg.Done() // garantisce il Done() anche in caso di panic
    // ... resto del worker ...
}()
```

### 1.2 [ALTA] Race Condition su `c.downloading` e campi correlati in `irc.Client`

**File:** `internal/irc/client.go`  
**Problema:** Il campo `downloading` (bool) è protetto da `c.mu` in alcuni punti ma non in tutti. Ad esempio:
- In `downloadPackAtIndex` (riga ~460): `c.downloading` viene letto senza lock prima del `waitForCurrentPack()`
- In `handleDCCSend` (handlers.go): `c.downloading = true` è sotto lock, ma `c.downloadError` è letto in `waitForCurrentPack` senza lock consistente

Inoltre, `c.packFilename` e `c.packSize` (aggiunti recentemente) sono letti in `downloadPackAtIndex` con lock, ma `parsePackInfoFromNotice` in `handlers.go` li scrive sotto lock — questo è corretto, ma il pattern di lock in tutto il file è incoerente.

**Fix consigliato:**
- Creare metodi getter/setter atomici o usare `atomic.Bool` per `downloading`
- Fare un audit completo di tutti gli accessi a `c.downloading`, `c.downloadError`, `c.packFilename`, `c.packSize`

## 2. Concorrenza e Goroutine — Issue MEDIA

### 2.1 [MEDIA] `sync.Once` reset manuale in `irc.Client.resetForPack()`

**File:** `internal/irc/client.go` (righe 437-438)  
**Problema:**
```go
c.closeOnce = sync.Once{}
c.startOnce = sync.Once{}
```

Questo è un **code smell pericoloso**: `sync.Once` è progettato per essere usato esattamente una volta per istanza. Resettarlo assegnando un valore zero è un pattern non documentato e potenzialmente unsafe in future versioni di Go. Se `resetForPack()` viene chiamato concorrentemente, due goroutine potrebbero osservare stati diversi di `sync.Once`.

**Fix consigliato:**
- Usare `atomic.Bool` o un canale `doneCh` per ogni ciclo di download
- Ogni pack dovrebbe avere il proprio stato di lifecycle, non condividere `sync.Once` a livello di client

```go
type packState struct {
    done    chan struct{}
    started chan struct{}
    once    sync.Once // per-pack, non per-client
}
```

---

### 2.2 [MEDIA] Channel non chiuso in `irc.Client.connectedCh`

**File:** `internal/irc/client.go` (riga ~133, 291)  
**Problema:** `connectedCh` viene ricreato in `resetForPack()` ma non viene mai chiuso esplicitamente in caso di errore di connessione. Se il client fallisce la connessione, il canale rimane aperto e `waitForConnection()` potrebbe bloccare indefinitamente.

**Fix consigliato:**
- Chiudere `connectedCh` nel cleanup del client o usare `context.Done()` come segnale di fallback

### 2.3 [MEDIA] `findBotOnConnectedServers` — context leak potenziale

**File:** `internal/api/handlers_search.go` (righe 530-625)  
**Problema:**
```go
ctx, cancel := context.WithTimeout(r.Context(), 15*time.Second)
defer cancel()
```

Questo è corretto, ma all'interno lancia goroutine che registrano handler su `girc.Client`. Se il context scade prima che le goroutine completino, i handler rimangono registrati sul client IRC (non vengono rimossi). `client.Handlers.Remove(cuid)` viene fatto solo nel flusso normale, ma se il context scade durante `whoisBotOnServer`, il `defer` chiude il canale `resultCh` e il `select` esce, ma i handler girc NON vengono rimossi.

**Fix consigliato:**
- Estrarre la logica di cleanup dei handler in un `defer` dedicato all'interno di `whoisBotOnServer`, oppure
- Usare un pattern con `doneCh` che garantisce cleanup anche su context cancellation

```go
func whoisBotOnServer(...) bool {
    // ...
    cuid := client.Handlers.Add(...)
    defer client.Handlers.Remove(cuid) // spostare QUI, subito dopo Add
    // ...
}
```

### 2.4 [MEDIA] `SQLiteStore` — `SetMaxOpenConns(1)` può causare deadlock nelle goroutine

**File:** `internal/store/sqlite.go` (riga ~37)  
**Problema:** `db.SetMaxOpenConns(1)` con WAL mode permette letture concorrenti ma una sola connessione aperta. Se una goroutine tiene una connessione aperta su una query lunga (es. `GetDownloadHistory` con molti filtri) e un'altra goroutine prova a fare una scrittura, la seconda potrebbe bloccare finché la connessione non viene rilasciata.

**Fix consigliato:**
- Aumentare a `SetMaxOpenConns(2)` o `3` per separare letture e scritture
- Documentare il trade-off: SQLite con WAL supporta letture concorrenti ma `SetMaxOpenConns(1)` le limita artificialmente

### 2.5 [MEDIA] `progressFn` callback in `queue.Manager` non protetto da lock

**File:** `internal/queue/manager.go` (riga ~665+)  
**Problema:** Il callback `progressFn` passato a `runDownload` chiama `qm.emitEvent()` e modifica lo stato del manager (`UpdateDownloadProgress`). Se `progressFn` viene chiamato frequentemente dal worker IRC (ogni blocco di dati DCC), e contemporaneamente `handleBulkDownloads` o `PauseDownload` modificano lo stato, si crea una race condition.

Sebbene `emitEvent` usi un lock interno (in `pubsub.PubSub`), le chiamate a `qm.store.UpdateDownloadProgress` NON sono protette dal lock del manager.

**Fix consigliato:**
- Aggiungere `qm.mu.RLock()`/`RUnlock()` attorno alla lettura dello stato e `qm.mu.Lock()` attorno alle modifiche
- Considerare un canale buffered per gli eventi di progress invece di callback diretti

## 3. Memory e Resource Leak

### 3.1 [MEDIA] `keepalive` ticker in `handleEvents` — ticker leak in caso di errore precoce

**File:** `internal/api/handlers_sse.go` (riga ~123)  
**Problema:**
```go
keepalive := time.NewTicker(30 * time.Second)
defer keepalive.Stop()
```

Questo è corretto, ma prima del `keepalive` ci sono molti punti di ritorno precoce (`writeError`, return) che NON arrivano al `defer keepalive.Stop()` se l'errore avviene prima della dichiarazione. In realtà il `defer` è posizionato correttamente, ma l'intero handler crea un nuovo ticker per ogni connessione SSE — con molti client connessi, questo è accettabile ma consuma una goroutine per ogni ticker.

**Fix consigliato:**
- Condividere un singolo ticker a livello di server SSE, oppure
- Usare `time.AfterFunc` per keepalive sporadici

---

### 3.2 [BASSA] `os.Remove(srcPath)` in `queue/worker.go` — errore ignorato

**File:** `internal/queue/worker.go` (riga ~178)  
**Problema:**
```go
os.Remove(srcPath) // dopo copyFile riuscito
```

L'errore di rimozione del file sorgente dopo copia cross-filesystem viene ignorato. Se il filesystem è pieno o corrotto, il file temporaneo rimane e occupa spazio.

**Fix consigliato:**
```go
if err := os.Remove(srcPath); err != nil {
    logger.Printf("warning: failed to remove temp file %s: %v", srcPath, err)
}
```

---

### 3.3 [BASSA] `os.Remove(destPath)` in `queue/worker.go` (riga ~168) — errore gestito ma incomplete

Il `os.Remove(destPath)` per `overwrite` policy è gestito correttamente, ma se il file è aperto da un altro processo, `os.Remove` fallirà silenziosamente su Windows (non su Unix). Documentare il comportamento cross-platform.

## 4. Bug Subdoli di Concorrenza

### 4.1 [MEDIA] Loop variable captured in goroutine — `searchagg/aggregator.go`

**File:** `internal/searchagg/aggregator.go` (riga ~290)  
**Problema:**
```go
for _, p := range providers {
    go func(name string) {
        // usa p
    }(p.Name)
}
```

Anche se il parametro `name` è passato correttamente, se `p` viene usato DENTRO la goroutine (non solo `name`), c'è il classico bug di capture della loop variable. Verificare che dentro la goroutine non si acceda a `p` direttamente.

**Fix consigliato:**
- Passare l'intero provider per valore, non per reference
- O copiare tutti i campi necessari in variabili locali prima della `go func`

---

### 4.2 [BASSA] `scanDownloadFromRows` — `return nil, nil` su errore di scan silenzia dati corrotti

**File:** `internal/store/sqlite.go` (riga ~1032)  
**Problema:**
```go
if err := rows.Scan(...); err != nil {
    return nil, nil // Silently skip corrupted row
}
```

Una riga corrotta viene silenziosamente saltata senza log. Questo può nascondere bug di schema o dati corrotti.

**Fix consigliato:**
```go
if err := rows.Scan(...); err != nil {
    // Log the corrupted row for debugging
    return nil, fmt.Errorf("scanning download row: %w", err)
}
```
Se lo skip intenzionale è voluto, almeno loggare a livello Warning.

## 5. Code Quality e Smell

### 5.1 [BASSA] Funzione `parsePackInfoFromNotice` con regex ricompilate ad ogni chiamata

**File:** `internal/irc/handlers.go` (riga ~247)  
**Problema:**
```go
packFileRe := regexp.MustCompile(`pack #\d+\s*\(\s*"([^"]+)"\s*\)`)
```

La regex viene compilata ad ogni chiamata della funzione, che avviene per ogni NOTICE dal bot. Questo è overhead inutile.

**Fix consigliato:**
```go
var (
    packFileRe    = regexp.MustCompile(`pack #\d+\s*\(\s*"([^"]+)"\s*\)`)
    sizePatterns  = []*regexp.Regexp{
        regexp.MustCompile(`(?i)grandezza\s*([\d.]+)\s*(gb|mb|kb)`),
        regexp.MustCompile(`(?i)size\s*:?\s*([\d.]+)\s*(gb|mb|kb)`),
        regexp.MustCompile(`(?i)\(([\d.]+)\s*(gb|mb|kb)\)`),
    }
)
```

**File:** `internal/api/api.go`

### 5.3 [BASSA] `stripIRCFormatting` — allocazione di `[]rune(s)` duplicata

**File:** `internal/ircmanager/utils.go` (riga ~71)  
**Problema:** `runes := []rune(s)` crea una copia dell'intera stringa, poi itera. Inoltre `strings.ToValidUTF8` è già chiamata prima, quindi la stringa è già valida.

**Fix consigliato:**
- Iterare direttamente sulla stringa con `for i := 0; i < len(s);` e gestire i byte
- O usare `utf8.DecodeRuneInString` per iterare senza allocare `[]rune`

results := client.DownloadAll()

## 6. Best Practices Go

### 6.1 [BASSA] `err != nil` check duplicato in `scanWatchlist`

**File:** `internal/store/sqlite.go` (righe ~1083-1090)  
**Problema:**
```go
if err != nil {
    if err == sql.ErrNoRows {
        return nil, nil
    }
    return nil, fmt.Errorf("scanning watchlist: %w", err)
}
// ...
if err != nil {  // <-- err non è stato riassegnato, questo check è dead code
    if err == sql.ErrNoRows {
        return nil, nil
    }
    return nil, fmt.Errorf("scanning watchlist: %w", err)
}
```

Il secondo blocco `if err != nil` è dead code perché `err` non è stato modificato dopo il primo check.

**Fix consigliato:**
```go
// Rimuovere il blocco duplicato
```

---

### 6.2 [BASSA] `levelWriter.Write` accumula in `buf` senza limite

**File:** `internal/logging/logging.go` (riga ~215)  
**Problema:** `levelWriter` accumula byte in `w.buf` finché non trova un newline. Se un writer produce output senza mai newline (molto raro ma possibile), `w.buf` cresce indefinitamente.

**Fix consigliato:**
```go
const maxBufSize = 64 * 1024

func (w *levelWriter) Write(p []byte) (int, error) {
    if len(w.buf)+len(p) > maxBufSize {
        // Flush entire buffer as a single line to prevent unbounded growth
        w.l.log(w.level, string(w.buf))
        w.buf = w.buf[:0]
    }
    // ... resto
}
```

### 6.3 [BASSA] `UpdateDownloadProgress` ritorna `err` senza wrapping

**File:** `internal/store/sqlite.go` (riga ~390)  
**Problema:**
```go
func (s *SQLiteStore) UpdateDownloadProgress(...) error {
    _, err := s.db.Exec(...)
    return err  // ritorna raw error
}
```

Non è wrapped con contesto.

**Fix consigliato:**
```go
if err != nil {
    return fmt.Errorf("updating download progress for id %d: %w", id, err)
}
return nil
```

## 7. Performance

### 7.1 [BASSA] `GetDownloadHistory` — query SQL dinamica non usa prepared statements

**File:** `internal/store/sqlite.go` (riga ~490+)  
**Problema:** La query dinamica viene costruita con concatenazione di stringhe per i `whereClauses`, anche se i valori usano `?` placeholder. Ogni combinazione di filtri produce una query SQL diversa, invalidando il query plan cache di SQLite.

**Fix consigliato:**
- Usare una query statica con condizioni `AND (?1 = '' OR filename LIKE ?2)` per permettere a SQLite di riutilizzare il query plan
- O accettare il trade-off dato il basso volume di query atteso

### 7.2 [BASSA] `eventsSince` in `sse.Hub` — scansione lineare O(n)

**File:** `internal/sse/hub.go`  
**Problema:** `EventsSince(lastEventID)` scorre linearmente tutti gli eventi bufferizzati. Con buffer grande e molti reconnect, è O(n) per ogni client.

**Fix consigliato:**
- Usare un indice ad albero o mantenere un map[eventID]*Event per lookup O(1)
- Oppure usare un ring buffer con puntatori di inizio/fine

## 9. Suggerimenti Architetturali

### 9.1 Separare `irc.Client` in lifecycle più piccoli
Attualmente `irc.Client` gestisce molti pack in sequenza con stato condiviso (`sync.Once`, canali). Considerare di creare un `PackDownloader` per ogni pack, con il proprio context e canali, e usare `irc.Client` solo come factory di connessioni.

### 9.2 Aggiungere `go test -race` alla CI
Il progetto ha già test (visti `*_test.go`), ma raccomando fortemente di aggiungere `go test -race ./...` alla pipeline CI. Questo avrebbe catturato la race condition su `downloading` e `progressFn`.

### 9.3 Usare `context.WithoutCancel` o propagazione esplicita
In alcuni punti il context viene usato per timeout corti (es. WHOIS 15s) ma il context padre (HTTP request) può cancellarsi prima, causando comportamenti non deterministici. Valutare se il timeout debba essere indipendente dal context HTTP.
