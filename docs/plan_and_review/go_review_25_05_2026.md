## 1. Concorrenza e Goroutine

### 1.1 — SSE Hub: race tra Publish e Unsubscribe su canali chiusi
**Severità:** ALTA  
**File:** `internal/sse/hub.go` — metodo `Publish()`

**Problema:**  
`Publish()` fa uno snapshot dei client sotto `h.mu.Lock()`, rilascia il lock, e poi itera sullo snapshot inviando eventi. Durante questa iterazione, una `Unsubscribe()` concorrente può rimuovere e chiudere un canale. Il `defer/recover` cattura il panic del `send on closed channel`, ma è un workaround, non una soluzione strutturale.

```go
// hub.go: Publish() — lo snapshot clients viene iterato senza lock
clients := make([]chan Event, 0, len(h.clients))
for ch := range h.clients {
    clients = append(clients, ch)
}
h.mu.Unlock()

// Qui Unsubscribe concorrente può close(ch)
for _, ch := range clients {
    func() {
        defer func() { recover() }() // workaround fragile
        select {
        case ch <- evt:
        default:
            dead = append(dead, ch)
        }
    }()
}
```

**Fix consigliato:**
L'approccio attuale con `recover` è pragmatico ma fragile. Una soluzione più robusta è usare un tipo di canale "safe" con un flag atomico `closed`, oppure mantenere il lock durante l'invio (accettando il costo). In alternativa, si può fare l'eviction dei client lenti con un meccanismo di `TrySend` su canale.

### 1.2 — Queue Manager: tryDispatch chiamato da più goroutine senza protezione di rientro
**Severità:** MEDIA  
**File:** `internal/queue/manager.go`

**Problema:**  
`tryDispatch()` può essere chiamata da:
- `Enqueue()` (goroutine HTTP handler)
- `monitorLoop()` (goroutine periodica)
- `completeFn` callback (goroutine di download)
- callback disk space recovery (goroutine diskmon)
- `ResumeDownload()`, `PauseDownload()`, `RemoveDownload()`

Non c'è alcun meccanismo di mutua esclusione su `tryDispatch`. Due chiamate concorrenti potrebbero entrambe superare i controlli di `activeCount` e `channelSlots`, causando l'avvio di download che eccedono i limiti configurati (es. 2 download sullo stesso canale).

```go
// manager.go — tryDispatch() non è protetta da lock
func (qm *QueueManager) tryDispatch() {
    // ...
    qm.mu.RLock()
    activeCount := qm.globalCount
    qm.mu.RUnlock()
    // GAP: un'altra goroutine può aver incrementato globalCount qui
    if activeCount >= maxParallel {
        return
    }
    // ...
    qm.startDownload(d) // può essere chiamata due volte per lo stesso slot
}
```

**Fix consigliato:**
Aggiungere un `sync.Mutex` dedicato (`dispatchMu`) acquisito all'inizio di `tryDispatch()`, oppure usare un pattern try-lock con `select` su un canale da 1 per garantire singola esecuzione alla volta.

### 1.3 — IRC Client: mix di sync.Mutex e atomic in stato condiviso
**Severità:** MEDIA  
**File:** `internal/irc/client.go`

**Problema:**  
Il `Client` usa sia `c.mu sync.Mutex` che variabili `atomic.*` per proteggere lo stesso stato logico, creando potenziali race condition sottili.

Esempi:
- `c.progress` è `int64` modificato con `atomic.AddInt64` in `receiveData()` ma letto sotto `c.mu.Lock()` in `finishSuccess()` — inconsistente.
- `c.downloading` è `bool` modificato sotto `c.mu` in `startDownload()` ma letto sotto `c.mu` in `progressPrinter()` — ok, ma l'atomic su `progress` crea comunque una race con `filesize` (protetto da mu).
- `c.dccConn`, `c.dccFile` acceduti in `receiveData()` SENZA lock (linea `c.dccConn.Read(buf)` e `c.dccFile.Write(buf[:n])` — usano il valore raw catturato dalla closure, ma `receiveData` viene spawnata da `startDownload` che setta `dccConn` e `dccFile` sotto lock prima di avviare la goroutine. Funziona ma è fragile.

**Fix consigliato:**
Rendere coerente la protezione: o tutto atomico o tutto mutex. Per `progress`, usare `atomic.LoadInt64` ovunque (come già fatto in `enqueueACK`), non `c.mu.Lock()`.

### 1.4 — PubSub Hub: Publish tiene RLock mentre invia a subscriber
**Severità:** BASSA/MEDIA  
**File:** `internal/pubsub/pubsub.go`

**Problema:**  
`Publish()` acquisisce `RLock`, itera sui subscriber e invia eventi con `select`. Se un subscriber è lento, il `default` scarta l'evento — ok, non blocca. Tuttavia, se un subscriber, in un'altra goroutine, tenta di chiamare `Unsubscribe()` mentre `Publish()` è in esecuzione, si ha una race: `Unsubscribe()` tenta di acquisire il write lock, ma `Publish()` tiene il read lock.

Questo non causa deadlock grazie a `sync.RWMutex` (un RLock non blocca altri RLock), ma se ci sono molte chiamate `Publish()` veloci, un `Unsubscribe()` può essere ritardato indefinitamente (writer starvation).

**Fix consigliato:**
Valutare se la contention è reale. Se sì, usare `sync.Map` o un pattern copy-on-write (snapshot della slice sotto lock, poi iterazione senza lock) come fa già l'SSE Hub.

### 1.5 — IRC Client: goroutine orphan in connect() su errore
**Severità:** BASSA  
**File:** `internal/irc/client.go` — metodo `connect()`

**Problema:**
```go
go func() { c.ircErrCh <- c.irc.Connect() }()
```
Se il `select` successivo va in timeout o cancellazione, `c.irc.Close()` viene chiamato, ma `c.irc.Connect()` potrebbe essere ancora in esecuzione (specie durante DNS resolution). La goroutine resterà in vita fino al completamento di `Connect()`, che potrebbe richiedere secondi. Il canale `ircErrCh` è bufferizzato (size 1), quindi la goroutine non blocca — ma è uno spreco di risorse.

**Fix consigliato:**
Nessun fix urgente (il buffer size 1 gestisce il caso). Per sicurezza, aggiungere un timeout al `Connect()` o usare `context.Context` con `Dialer`.

### 1.6 — IRC Manager: goroutine in managedConnection.connect() non tracciata
**Severità:** BASSA  
**File:** `internal/ircmanager/manager.go`

**Problema:**
```go
go func() {
    err := client.Connect()
    disconnected <- err
    close(disconnected)
}()
```
Questa goroutine non è tracciata da `mc.wg`. Se `mc.ctx` viene cancellato, `client.Close()` viene chiamato e il canale `disconnected` viene drenato col timeout. Tuttavia, se `client.Close()` fallisce o blocca, la goroutine rimane attiva indefinitamente.

**Fix consigliato:**
Tracciare anche questa goroutine con `mc.wg.Add(1)`/`mc.wg.Done()`, oppure usare un `context.Context` passato a `girc.Client`.

## 2. Bug Subdoli di Concorrenza

### 2.1 — IRC Client: finishWithError race condition su downloadError
**Severità:** ALTA  
**File:** `internal/irc/client.go` — metodo `finishWithError()`

**Problema:**
```go
func (c *Client) finishWithError(err error) {
    c.mu.Lock()
    if c.downloadError == nil {
        c.downloadError = err
    }
    c.mu.Unlock()
    if c.downloadDoneClosed.CompareAndSwap(false, true) {
        if c.downloadDone != nil {
            close(c.downloadDone)
        }
    }
}
```

C'è una race condition tra due chiamate concorrenti a `finishWithError()`:
1. Goroutine A: `c.mu.Lock()` → `downloadError == nil` → setta `downloadError`
2. Goroutine A: `c.mu.Unlock()`
3. Goroutine B: `c.mu.Lock()` → `downloadError != nil` → non sovrascrive
4. Goroutine B: `c.mu.Unlock()`
5. Goroutine A: `CompareAndSwap(false, true)` → `true`, chiude `downloadDone`
6. Goroutine B: `CompareAndSwap(false, true)` → `false`, NON chiude `downloadDone`

Questo va bene perché solo una goroutine chiuderà il canale. TUTTAVIA, se la goroutine B legge `downloadError` DOPO la unlock di A ma PRIMA che A chiuda il canale, B potrebbe restituire `downloadError` corretto ma senza aver chiuso il canale (che verrà chiuso da A). Funziona, ma è fragile.

**Fix consigliato:**
La logica è sostanzialmente corretta (CompareAndSwap funge da sync.Once). Documentare la sequenza con un commento esplicito.

### 2.2 — resetForPack: race con goroutine di download pendenti
**Severità:** ALTA  
**File:** `internal/irc/client.go` — metodo `resetForPack()`

**Problema:**  
`resetForPack()` chiude `c.downloadDone` (se non già chiuso), crea nuovi canali, e resetta lo stato. Tuttavia, le goroutine del pack precedente (`ackSender`, `progressPrinter`, `stallWatcher`) potrebbero ancora star leggendo `c.dccConn` o `c.downloading`:

```go
// ackSender - ancora in esecuzione su vecchio dccConn
case ack := <-c.ackQueue:  // c.ackQueue è stato sostituito!
```

Dopo `resetForPack()`, il vecchio `c.ackQueue` viene garbage-collectato. Le goroutine del pack precedente che stavano facendo `<-c.ackQueue` riceveranno un `receive on closed channel` (zero value), causando potenzialmente comportamenti errati.

L'`ackSender` del nuovo pack farà `range` sul NUOVO `c.ackQueue`, mentre il vecchio `ackSender` è ancora in esecuzione e tenta di leggere dal canale vecchio (ormai garbage). Non è tecnicamente una race condition se il vecchio `ackSender` è già uscito dal `select` tramite `c.downloadDone` (che è stato chiuso). Ma se il vecchio `ackSender` è bloccato in `conn.Write(ack)`...

**Fix consigliato:**
La chiusura di `downloadDone` viene fatta PRIMA di creare nuovi canali. Le goroutine (`ackSender`, `progressPrinter`, `stallWatcher`) ascoltano `downloadDone` e dovrebbero uscire. Il codice sembra corretto MA va verificato che `c.dccConn.Close()` in `stallWatcher` non causi race con `receiveData()`.

### 2.3 — ircmanager: ConnectServer — defer delete(connecting) dopo Unlock
**Severità:** BASSA  
**File:** `internal/ircmanager/manager.go` — metodo `ConnectServer()`

**Problema:**
```go
m.mu.Unlock()
// ... operazioni senza lock ...
defer func() {
    m.mu.Lock()
    delete(m.connecting, srv.ID)
    m.mu.Unlock()
}()
```

Il `defer` è registrato DOPO aver rilasciato il lock. Se una panic avviene tra `Unlock()` e il `defer`, `delete(m.connecting, srv.ID)` non verrà mai eseguito, e il server rimarrà permanentemente in stato "connecting" per il sistema di prevenzione duplicati.

**Fix consigliato:**
Spostare il `defer` subito dopo aver marcato `m.connecting[srv.ID]` e PRIMA della `Unlock()`.

### 2.4 — scanDownloadFromRows: nil di ritorno buca i conteggi
**Severità:** BASSA  
**File:** `internal/store/sqlite.go` — funzione `scanDownloadFromRows()`

**Problema:**
```go
func scanDownloadFromRows(rows *sql.Rows) (*DownloadRecord, error) {
    // ...
    if err := rows.Scan(...); err != nil {
        log.Printf("WARNING: skipping corrupted download row (scan error: %v)", err)
        return nil, nil // ⟵ ritorna nil, nil per segnalare "skip"
    }
}
```

Il chiamante `scanDownloads()` tratta `nil, nil` come "nessun record, continua senza errori":
```go
if d != nil {
    downloads = append(downloads, *d)
}
```

Questo è un pattern insidioso: il chiamante ignora silenziosamente i record corrotti, ma la paginazione e il conteggio totale (ottenuti con COUNT separato) saranno inconsistenti. L'API di history restituirà meno risultati di quelli dichiarati dal conteggio.

**Fix consigliato:**
Aggiungere un errore sentinella (`ErrCorruptedRow`) e gestirlo esplicitamente nel chiamante.

## 3. Memory e Resource Leak

### 3.1 — DiskMon: Available() restituisce sempre 0
**Severità:** MEDIA  
**File:** `internal/diskmon/monitor.go`

**Problema:**
```go
func (m *Monitor) Available() int64 {
    m.mu.RLock()
    defer m.mu.RUnlock()
    // Since we don't cache available, call Check() for fresh data
    return 0  // ⟵ SEMPRE 0. Il commento dice di chiamare Check() ma non lo fa.
}
```

Questo metodo è un getter che non funziona. Qualsiasi chiamante che usa `Available()` per prendere decisioni riceverà 0, che potrebbe causare falsi positivi di "disco pieno" o comportamenti errati.

**Fix consigliato:**
Aggiungere un campo `available int64` cachato, aggiornato da `Check()`, e restituirlo in `Available()`.

### 3.2 — IRC Client: dccConn e dccFile in receiveData senza protezione
**Severità:** MEDIA  
**File:** `internal/irc/client.go` — metodo `receiveData()`

**Problema:**
```go
func (c *Client) receiveData() {
    defer func() {
        c.mu.Lock()
        c.downloading = false
        if c.dccFile != nil {
            c.dccFile.Close()
        }
        // ... dccConn NON viene chiuso qui!
        c.mu.Unlock()
    }()
    // ...
    for {
        n, err := c.dccConn.Read(buf)  // accesso NON protetto a dccConn
        // ...
        c.mu.Lock()
        _, werr := c.dccFile.Write(buf[:n])  // accesso protetto a dccFile
        c.mu.Unlock()
    }
}
```

`c.dccConn` viene letto senza lock, e non viene chiuso nel `defer`. Viene chiuso solo da `stallWatcher` o da `resetForPack`. Se `receiveData` esce per un errore di read (connessione persa), `dccConn` rimane aperto fino al prossimo `resetForPack`.

**Fix consigliato:**
Chiudere `c.dccConn` nel `defer` di `receiveData()` sotto lock.

---

### 3.3 — Queue Manager: cleanup goroutines alla Stop()
**Severità:** BASSA  
**File:** `internal/queue/manager.go` — metodo `Stop()`

**Problema:**
```go
func (qm *QueueManager) Stop() {
    if qm.stopDiskCheck != nil {
        qm.stopDiskCheck()
        <-qm.diskCheckDone
    }
    qm.cancel()
    <-qm.done
    // ...
}
```

Il `qm.subscriber` (pubsub.Hub) non viene mai chiuso. I subscriber (SSE handlers) che stanno ancora ascoltando continueranno ad attendere indefinitamente su canali che non verranno mai più alimentati.

**Fix consigliato:**
Aggiungere un metodo `Close()` a `pubsub.Hub` e chiamarlo in `Stop()` per chiudere tutti i canali dei subscriber.

### 3.4 — IRC Client: time.After leak nella logica throttling
**Severità:** BASSA  
**File:** `internal/irc/client.go` — metodo `receiveData()`

**Problema:**
```go
case <-time.After(time.Duration(sleepTime * float64(time.Second))):
```

All'interno del loop di `receiveData`, ogni chunk throttled crea un nuovo `time.After`. Se il download è lungo (es. 10GB a 1MB/s = ~10.000 secondi), vengono creati migliaia di timer. `time.After` non viene garbage collectato fino allo scadere del timer, causando un accumulo di risorse.

**Fix consigliato:**
Usare `time.NewTimer` fuori dal loop e resettarlo con `timer.Reset()`.

## 5. Best Practices Go

### 5.1 — Uso non idiomatico di sync.Once per canali
**Severità:** BASSA  
**File:** `internal/irc/client.go`

**Problema:**
```go
c.downloadDoneClosed    atomic.Bool
c.downloadStartedClosed atomic.Bool
```
Questi due `atomic.Bool` sono usati esattamente come `sync.Once` (garantire chiusura singola del canale). L'uso di `atomic.Bool` + `CompareAndSwap` è meno idiomatico di `sync.Once` e più soggetto a errori.

**Fix consigliato:**
Sostituire con `sync.Once`:
```go
var downloadDoneOnce sync.Once
downloadDoneOnce.Do(func() { close(c.downloadDone) })
```

---

### 5.2 — Error handling: panic recovery senza stack trace
**Severità:** BASSA  
**File:** `internal/ircmanager/manager.go` — `run()`, `internal/sse/hub.go` — `Publish()`

**Problema:**  
I `recover()` catturano il panic ma non loggano lo stack trace, rendendo difficile il debugging:
```go
defer func() {
    if r := recover(); r != nil {
        mc.manager.logger.Printf("PANIC in run() for server %d: %v", mc.id, r)
        // ⟵ nessuno stack trace!
    }
}()
```

**Fix consigliato:**
```go
import "runtime/debug"
mc.manager.logger.Printf("PANIC in run() for server %d: %v\n%s", mc.id, r, debug.Stack())
```

### 5.3 — Variabili non utilizzate
**Severità:** BASSA  
**File:** `internal/sse/hub.go` — `Subscribe()`

**Problema:**
```go
clientCount := len(h.clients)
_ = clientCount  // ⟵ assegnato e mai usato
```

**Fix consigliato:**
Rimuovere la variabile o usarla per logging.

## 6. Performance

### 6.1 — SSE Hub: EventsSince fa scansione O(n)
**Severità:** BASSA  
**File:** `internal/sse/hub.go`

**Problema:**
`EventsSince(lastEventID)` itera su tutti gli eventi nel buffer per trovare quelli con ID > lastEventID. Con buffer di 200 eventi non è un problema, ma se il buffer crescesse sarebbe inefficiente. L'evento ha un ID monotonic, quindi si potrebbe calcolare l'offset direttamente:

```go
// Invece di iterare tutto il buffer:
offset := lastEventID - (h.nextID - int64(h.bufferCount))
```

**Fix consigliato:**
Sostituire il loop con accesso indicizzato diretto tramite calcolo dell'offset.

---

### 6.2 — Lock contention potenziale in pubsub.Publish
**Severità:** BASSA  
**File:** `internal/pubsub/pubsub.go`

**Problema:**  
`Publish()` itera su TUTTI i subscriber sotto `RLock`. Con molti subscriber e pubblicazioni frequenti (es. eventi di progress download ogni ~100ms con molti download paralleli), il lock può diventare un collo di bottiglia.

**Fix consigliato:**
Snapshottare la slice dei subscriber sotto lock (come fa SSE Hub), poi iterare senza lock.

---

### 6.3 — Allocazioni in scanDownload
**Severità:** BASSA  
**File:** `internal/store/sqlite.go`

**Problema:**  
`scanDownload` e `scanDownloadFromRows` allocano `sql.NullString` per ogni campo temporale. Con history queries che restituiscono centinaia di record, sono centinaia di allocazioni per `NullString` che potrebbero essere evitate usando `*time.Time` con `sql.NullTime` o scanner custom.

**Fix consigliato:**
Per query di history che restituiscono molti record, usare `sql.NullTime` (Go 1.22+) o uno scanner custom.

## Giudizio Complessivo

Il codebase è **di buona qualità complessiva**. L'architettura è solida, con una chiara separazione delle responsabilità (IRC, queue, SSE, store, API). L'uso di goroutine, channel, e context è generalmente corretto e ben pensato. La gestione degli errori è idiomatica.

Le aree di miglioramento principale sono:
1. **Protezione della concorrenza in `queue/manager.go`** (tryDispatch non thread-safe)
