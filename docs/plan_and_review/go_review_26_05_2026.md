## Sommario Esecutivo

Il progetto mostra una buona base architetturale (uso esteso di context, cleanup loop con ticker stoppati, separazione in package chiara). Tuttavia emergono alcuni rischi importanti in aree concorrenti ad alta pressione:

1. timeout wrapper con goroutine potenzialmente orfane in ricerca provider;
2. race/fragilita su publish/close di hub concorrenti (con recover come workaround);
3. lifecycle incompleto di timer callback in startup queue manager;
4. race su stato condiviso in callback WHOIS concorrenti;
5. alcuni punti di robustezza/error handling e performance facilmente migliorabili.

## 1) Concorrenza e Goroutine

### 1.1 Timeout wrapper con possibile goroutine leak in ricerca provider
- **Severita:** Alta
- **Riferimento:** `internal/searchagg/aggregator.go` (`searchLive`)
- **Problema:** per ogni provider viene avviata una goroutine che esegue `engine.Search(searchCtx, query)` e segnala `done`. Se scatta `searchCtx.Done()` prima del `done`, il branch timeout ritorna subito nel worker e la goroutine interna continua finche `engine.Search` non rientra. Se un provider ignora il context o si blocca su I/O, questa goroutine puo restare viva a lungo.
- **Perche e rischioso:** sotto carico (piu query parallele) si puo accumulare backlog di goroutine non terminate in tempo, con uso memoria/cpu e latenza imprevedibile.
- **Fix consigliato:** evitare la doppia goroutine quando possibile; chiamare direttamente `engine.Search` nel worker con context timeout e delegare la cancellazione al provider. Se serve wrapper, introdurre un guard-join breve post-timeout e metrica warning.

Esempio:
```go
searchCtx, cancel := context.WithTimeout(ctx, timeout)
defer cancel()

packs, err := engine.Search(searchCtx, query)
latency := time.Since(start)
if err != nil {
    results <- engineResult{name: name, err: err, latency: latency}
    return
}
results <- engineResult{name: name, packs: packs, latency: latency}
```

### 1.2 Timer startup queue non gestito nel lifecycle di stop
- **Severita:** Media
- **Riferimento:** `internal/queue/manager.go` (`Start`, `Stop`)
- **Problema:** `Start()` usa `time.AfterFunc(delay, ...)` senza conservare il `*time.Timer`; `Stop()` non puo cancellarlo. Se avviene shutdown prima dello scadere, il callback puo partire post-stop e chiamare `tryDispatch()` fuori dal lifecycle desiderato.
- **Perche e rischioso:** comportamento non deterministico in shutdown, callback tardive, difficile diagnosi in produzione.
- **Fix consigliato:** salvare il timer nel manager e stoppare in `Stop()`.

Esempio:
```go
// in Manager
startupTimer *time.Timer

// in Start
qm.startupTimer = time.AfterFunc(delay, func() {
    close(qm.startupReady)
    qm.tryDispatch()
})

// in Stop
if qm.startupTimer != nil {
    qm.startupTimer.Stop()
}
```

### 1.3 Goroutine fallback JOIN->XDCC per WHOIS con accumulo transitorio
- **Severita:** Media
- **Riferimento:** `internal/irc/handlers.go` (handler `RPL_ENDOFWHOIS`)
- **Problema:** per fallback viene creata una goroutine con `time.After(5s)`. In burst di richieste puo accumularsi un numero elevato di goroutine temporanee.
- **Perche e rischioso:** non e leak permanente, ma puo creare picchi evitabili di scheduling/memoria.
- **Fix consigliato:** usare timer riusabile per sessione/pack o centralizzare il fallback in state machine unica per client.

### 1.4 Race su flag `gotReply` in callback WHOIS concorrenti
- **Severita:** Alta
- **Riferimento:** `internal/api/handlers_search.go` (`whoisBotOnServer`)
- **Problema:** `gotReply` e un bool condiviso tra due callback (`RPL_WHOISUSER`, `ERR_NOSUCHNICK`) e il flow principale, senza sincronizzazione.
- **Perche e rischioso:** data race intermittente con comportamento non deterministico (duplicazioni/ordine race-dependent).
- **Fix consigliato:** usare `sync.Once`, `atomic.Bool` o solo canale con protocollo single-writer.

Esempio con `sync.Once`:
```go
var once sync.Once
send := func(v string) {
    once.Do(func() {
        select {
        case resultCh <- v:
        default:
        }
    })
}
```

## 2) Bug Subdoli di Concorrenza

### 2.1 TOCTOU su Publish/Close nel pubsub generico (panic mascherato)
- **Severita:** Media
- **Riferimento:** `internal/pubsub/pubsub.go` (`Publish`, `Close`)
- **Problema:** `Publish` fa snapshot subscribers, poi invia fuori lock; in parallelo `Close/Unsubscribe` puo chiudere un channel. Il codice usa `recover()` per mascherare `send on closed channel`.
- **Perche e rischioso:** il recover evita crash ma nasconde bug di ordering e rende il comportamento difficile da osservare/testare.
- **Fix consigliato:** serializzare close/publish in modo deterministico (approccio lock-hold come in `internal/sse/hub.go`) oppure protocollo di shutdown con stato atomico + rimozione subscribers prima di close.

### 2.2 Early-return con lavoro concorrente ancora in corso nella ricerca bot su server
- **Severita:** Media
- **Riferimento:** `internal/api/handlers_search.go` (`findBotOnConnectedServers`)
- **Problema:** la funzione ritorna al primo risultato, ma le altre goroutine WHOIS continuano fino a timeout/cancel.
- **Perche e rischioso:** overhead transitorio eccessivo in ambienti con molti server connessi.
- **Fix consigliato:** usare `context.WithCancel` interno e cancellarlo appena arriva il primo hit.

Esempio:
```go
ctx2, cancel := context.WithCancel(ctx)
defer cancel()
...
if found {
    cancel() // stop best-effort delle altre goroutine
}
```

### 2.3 Sequenza shutdown SSE/HTTP migliorabile
- **Severita:** Bassa
- **Riferimento:** `cmd/xdcc-server/main.go`, `internal/sse/hub.go`
- **Problema:** il codice chiude SSE hub prima della cancellazione globale richieste HTTP. E corretto nella maggior parte dei casi, ma puo produrre una finestra in cui handler sta ancora flushando su stream in chiusura.
- **Fix consigliato:** invertire ordine (cancel richieste -> shutdown HTTP -> close hub) o introdurre barrier esplicita sugli handler SSE attivi.

## 3) Memory e Resource Leak

### 3.1 Timer callback non cancellabile (startup delay)
- **Severita:** Media
- **Riferimento:** `internal/queue/manager.go`
- **Problema:** stesso punto di 1.2, qui dal lato leak/resource lifecycle.
- **Fix consigliato:** conservare e stoppare `*time.Timer`.

### 3.2 Body HTTP non chiuso esplicitamente nei POST JSON
- **Severita:** Bassa
- **Riferimento:** `internal/api/handlers_download.go`, `internal/api/handlers_search.go` (e handler analoghi)
- **Problema:** decode via `json.NewDecoder(r.Body)` senza `defer r.Body.Close()`.
- **Nota tecnica:** in server net/http il body viene in genere chiuso dal server al termine handler, quindi non e un leak grave; tuttavia la chiusura esplicita resta una best practice utile per rilascio tempestivo risorse e coerenza stilistica.
- **Fix consigliato:** aggiungere `defer r.Body.Close()` dopo validazione metodo/content-type.

### 3.3 stopWithTimeout usa `time.After` per ogni stop step
- **Severita:** Bassa
- **Riferimento:** `cmd/xdcc-server/main.go` (`stopWithTimeout`)
- **Problema:** `time.After` alloca timer per invocazione (non critico ma evitabile in percorsi frequenti).
- **Fix consigliato:** `time.NewTimer` + `defer timer.Stop()`.

## 4) Code Quality e Smell

### 4.1 `recover()` come meccanismo normale di controllo flusso
- **Severita:** Media
- **Riferimento:** `internal/pubsub/pubsub.go` (`Publish`)
- **Problema:** usare panic/recover per gestire race previste degrada leggibilita e affidabilita.
- **Fix consigliato:** protocollo di chiusura esplicito senza panic previsto.

### 4.2 Parsing numerico con `fmt.Sscanf` poco robusto
- **Severita:** Bassa
- **Riferimento:** `internal/api/api.go` (`parseID`, `parseInt`, `parseInt64`, `parsePageParams`)
- **Problema:** parsing permissivo, gestione errori minimale, semantica meno chiara.
- **Fix consigliato:** `strconv.ParseInt/ParseUint/Atoi` con error wrapping contestuale.

### 4.3 Errori di encode JSON ignorati
- **Severita:** Bassa
- **Riferimento:** `internal/api/api.go` (`writeError`, `writeJSON`)
- **Problema:** `json.NewEncoder(...).Encode(...)` ignora errore.
- **Fix consigliato:** loggare a livello debug/warn l’errore di scrittura (tipico client disconnect).

## 5) Best Practices Go

### 5.1 Normalizzazione cache key non Unicode-safe
- **Severita:** Media
- **Riferimento:** `internal/searchagg/cache.go` (`cacheKey`)
- **Problema:** iterazione su rune ma append su `byte(ch)`; caratteri non ASCII vengono troncati/corrotti.
- **Impatto:** cache miss inattesi su query con caratteri internazionali, deduplica incoerente.
- **Fix consigliato:** usare `strings.ToLower`, `strings.Fields`, `strings.Join` (rune-safe).

Esempio:
```go
func cacheKey(query string) string {
    parts := strings.Fields(strings.ToLower(query))
    return strings.Join(parts, " ")
}
```

### 5.2 Mappatura errori context in API search
- **Severita:** Media
- **Riferimento:** `internal/api/handlers_search.go` (`handleSearch`)
- **Problema:** qualsiasi errore viene mappato a 500; `context.DeadlineExceeded` e `context.Canceled` andrebbero distinti.
- **Fix consigliato:** 503/504 per timeout, 499/408-like handling per cancel lato client.

### 5.3 Query DB senza `Context` nelle store API
- **Severita:** Media
- **Riferimento:** `internal/store/sqlite.go` (metodi `Query/Exec` senza varianti `Context`)
- **Problema:** cancellazioni request non propagate al layer DB.
- **Fix consigliato:** introdurre gradualmente `QueryContext/ExecContext` nei path hot.

## 6) Performance

### 6.1 Pool SQLite probabilmente stretto sotto carico misto read/write
- **Severita:** Media
- **Riferimento:** `internal/store/sqlite.go` (`NewSQLiteStore`)
- **Problema:** `SetMaxOpenConns(3)` puo diventare collo di bottiglia con ricerca parallela + update stats + cache persistence.
- **Fix consigliato:** benchmarkare e valutare `MaxOpenConns` piu alto o batching asincrono delle scritture di stats.

### 6.2 Scrittura stats provider sincrona per ogni risultato
- **Severita:** Media
- **Riferimento:** `internal/searchagg/aggregator.go` (`recordProviderResult`)
- **Problema:** ogni risultato fa write DB immediata, aggiungendo latenza e contention su SQLite.
- **Fix consigliato:** coda buffered + worker di flush periodico (batch per finestra temporale).

### 6.3 Rotazione log con `os.Stat` ad ogni write
- **Severita:** Bassa
- **Riferimento:** `internal/logging/logging.go` (`log`, `rotateIfNeeded`)
- **Problema:** check size per ogni log line puo pesare ad alto throughput.
- **Fix consigliato:** throttling del check (es. ogni N linee o ogni 100-250ms).

## Osservazioni Positive

- `internal/sse/hub.go` mostra un disegno piu solido rispetto al pubsub generico: publish sotto lock, eviction client lenti non bloccante.
- uso diffuso di `context.WithTimeout` nei path critici.
- presenza di shutdown ordinato con timeout multipli in `cmd/xdcc-server/main.go`.
- miglioramenti gia presenti su `queue.Manager` (`dispatchMu`) per evitare dispatch concorrenti multipli.

## Suggerimenti Architetturali (Priorita)

1. **Priorita Alta (quick wins, 1 sprint):**
- rifattorizzare `searchLive` eliminando wrapper goroutine timeout fragile;
- rendere race-free `whoisBotOnServer` (`sync.Once`/atomics);
- introdurre timer lifecycle esplicito in queue startup delay;
- sostituire `recover` in `pubsub.Publish` con protocollo deterministico.

2. **Priorita Media (2 sprint):**
- introdurre propagazione `context` nel layer store per path hot;
- differenziare error mapping HTTP per timeout/cancel;
- rendere Unicode-safe `cacheKey`.

3. **Priorita Performance (iterativa):**
- batching async per provider stats;
- tuning pool SQLite guidato da metriche;
- ridurre overhead rotazione log per write burst.

## Piano di Hardening Consigliato

- numero goroutine per endpoint/path;
- timeout provider per motore;
- durata shutdown per componente;
- queue depth per write DB stats.

2. Aggiungere test mirati (in una fase successiva, non eseguiti in questa review):
- test race su `whoisBotOnServer`;
- test concorrenza su `pubsub.Publish/Close/Unsubscribe`;
- test cancellazione anticipata su `findBotOnConnectedServers`;
- test Unicode cache key.

3. Introdurre checklist PR concorrente:
- ownership channel close;
- no panic/recover come controllo flusso;
- context propagation end-to-end;
- cleanup esplicito timer/ticker/handler lifecycle.
