  Rationale:
  - 002 and 005 are pure documentation/build changes; safest to land first.

    CI migration test runs against the new schema before later
    plans touch DB code.
  - 001 is a one-line concurrency fix.
  - 004 is the largest of the S-effort plans (frontend + backend) but
    well-scoped.
  - 006 has the broadest existing-test blast radius; landing last
    means any other broken-test noise is resolved first.

## Findings considered and rejected

- **`agent.md` is large (34KB)** — not a finding; it's intentional AI
  context. Not actionable.
- **Migration to a different SQLite driver** — speculative; current
  `modernc.org/sqlite` (pure Go, CGO_ENABLED=0) is by-design per
  `CLAUDE.md`. Not worth doing.
- **Replacing `chi` router with stdlib `net/http` mux** — speculative;
  chi is idiomatic for this codebase and well-tested.
- **Finding F15 originally suggested "wire watchlist notifications"** —
  audit recon was incomplete; investigation for Plan 006 confirmed the
  wiring already exists end-to-end (schema, model, API, notifier
  providers, SSE event type, dispatcher callback). The remaining gap is
  the dispatcher ignoring `notify_enabled`, which is the actual scope
  of Plan 006.
- **Adding `girc` pinned-version commit hash to a vendor directory** —
  speculative dependency-management change; `go.mod` comment is
  sufficient for now.
- **Bulk import/export for presets/watchlists (DIR-D1)** — listed as a
  direction suggestion in `.kimchi/docs/audit-findings.md` but not
  selected for plan creation (lower priority; the codebase already has
  full-DB export/import).

## How to use these plans

- Each plan file is fully self-contained: an executor with zero
  context can run it end-to-end.
- Each plan has a **Drift check** at the top — run it first; if any
  in-scope file changed since this batch was written, compare the
  "Current state" excerpts against the live code before proceeding.
- Each plan has explicit **Scope** (in/out) and **STOP conditions**
  to keep executors from drifting into adjacent work.
- When an executor lands a plan, it updates the status row above to
  one of: `DONE`, `BLOCKED`, `REJECTED` (with one-line reason).
- If a plan is `BLOCKED`, the next round should refine it (don't
  silently drop the work).
